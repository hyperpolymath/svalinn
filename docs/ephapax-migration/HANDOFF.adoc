= Svalinn → Ephapax Migration — Handoff Charter
:toc: macro
:sectnums:
:sectnumlevels: 3

This is a *self-contained* charter for a delegated Claude Code session that will
replace svalinn's ReScript with *Ephapax*. It was disposed from the Ochránce
proof thread (provenance: `ochrance/docs/PROOFS.adoc`, Decision *D3* — *restated
in full below*; you do *not* need the ochrance repo in scope). Read it top to
bottom before doing anything: the first task is a viability *spike*, not the
port. Nothing here depends on chat memory.

toc::[]

== Role

You are picking up the *svalinn → Ephapax migration*. Your job is *not* to write
proofs. It is to migrate svalinn's ReScript to Ephapax and, before any porting,
to determine whether that migration is even viable today. The estate rule is
*migrate-then-prove*: proofs come after the language is final, because proving
the about-to-be-deleted ReScript is waste.

== Precondition — repository scope

This session *MUST* have both `hyperpolymath/svalinn` *and* `hyperpolymath/ephapax`
in repository scope. The root task (the readiness spike) reads the ephapax repo.

* If `ephapax` is *not* in scope: *STOP* and add it (use `list_repos` /
  `add_repo` if available, otherwise have the session re-created with both
  repos). Do not proceed on svalinn alone — that is exactly the wall that forced
  this hand-off.

== Decision on record (D3)

svalinn migrates ReScript → *Ephapax*. Ephapax replaces *all* the ReScript.
*AffineScript is not the target* — it surfaced only as a format exemplar for the
migration map. The migration discharges svalinn's security debt *structurally*
(typed boundary decoders + linear exactly-once tokens), not by patching ReScript.

== Deliverables (strictly in order)

. *Readiness spike* — gates everything below (Stage 0).
. *Migration map* — `docs/ephapax-migration/BLOCKER-LINEAGE.adoc` (Stage 1).
. *The port* — only if the spike is green (Stage 2).

== Stage 0 — Readiness spike

This is the one thing the originating session could not do (ephapax was out of
its scope). *Root question:* is Ephapax an implementable _application_ language
today, or still a proof-level calculus? svalinn is a network HTTP gateway; it
needs a real runtime.

=== Watch-for — D3 may be invalid

[WARNING]
====
If a *MUST*-have is *ABSENT with no FFI path*, the migration cannot proceed and
*Decision D3 itself is in question*. *STOP. Do not start the port.* Surface to
the user with the capability matrix and the live alternatives:

. stay on ReScript and prove it directly;
. reconsider *AffineScript* (general application language, typed-wasm, OCaml
  compiler) as the migration target instead;
. wait on Ephapax and defer the migration.

This is a high-information result — better learned from a cheap spike than
mid-port. Do not improvise a different target language without the user's
decision.

== Stage 1 — Migration map

Write `docs/ephapax-migration/BLOCKER-LINEAGE.adoc` (AsciiDoc, SPDX
`PMPL-1.0 OR MPL-2.0`). Follow the estate *Critical Chain* methodology
(`.machine_readable/6a2/methodology.a2ml`): Phase-0 finds the critical chain;
classify into rings; weight MUST(3×) / SHOULD(2×) / COULD(1×); convergent budget
structural70 / corrective20 / perfective10. The representation format is *fully specified below* (ring-0 critical chain /

A worked example of this exact format lives as the analogous
`BLOCKER-LINEAGE.adoc` in the `affinescript` repo — consult it *only if that repo
is already in your scope*; otherwise do not chase it, the structure here suffices.

Required sections:

* *State now* — ReScript, ~7,050 LOC, hazards (Appendix B), CI red (Appendix C).
* *State needed* — Ephapax gateway meeting the MVP (Appendix A), zero
  `Obj.magic`, JWT revocation true-by-construction.
* *Critical chain (ring-0)*, dependency-ordered, rooted at the spike:

Ephapax readiness → toolchain + CI green → core boundary decoders (kill

  benchmarks, JWT-parser fuzz harness.

== Stage 2 — The port

Only if the spike is green. Port the ≈27 canonical `src/` modules + the `ui/`
frontend (Appendix F), ReScript → Ephapax. Make the security wins *structural*,
not patched:

* Typed boundary decoders ⇒ the 20+ `Obj.magic` casts (Appendix B) become
  *impossible*, not merely discouraged.
* Ephapax *linear / exactly-once* tokens ⇒ JWT/JTI single-use + revocation
  ledger (kills `hasRevocationList = false`), OAuth nonce/PKCE single-use,
  session/container lifecycle (exactly-once start/stop), vörðr delegation.
* Ensure the Ephapax test suite *runs in CI*. The ReScript tests never executed
  under Deno (`@rescript/core` resolution); they vanish with the migration — do
  not port the breakage.

After migration, discharge the language-agnostic *specs* (Appendix E).

== Appendix A — MVP target

Source: `MVP.adoc`.

* `POST /verify` — validate a bundle against trust store + policy.
* `POST /run` — schedule + start a container *after* verification.
* JSON responses use `error-response.v1.json`.
* Gatekeeper policy — implement schema `spec/schemas/gatekeeper-policy.v1.json`;
  enforce required predicate types, allowed signer key IDs, log quorum = 1.
* Single-node *FIFO* scheduling; status: `queued | starting | running | failed |
  stopped`.
* vörðr delegation — translate request to vörðr CLI/API; capture audit metadata.
* NOTE: `MVP.adoc` references prototype tools `tools/mvp/svalinn_gateway.ts` (the
  one *approved* TypeScript exemption) and `svalinn_gate.py` — MVP scaffolding,
  not the production gateway being migrated. Leave the exemption until svalinn
  reaches production tier (its documented unblock condition).
* Deferred to 1.0: multi-log quorum, threshold sigs, multi-node, authn/authz,
  rate limiting.

== Appendix B — Hazards / port targets

Source: `PROOF-NEEDS.md`. 20+ `Obj.magic` casts in security paths:

* `src/auth/AuthMiddleware.res` — `Obj.magic(payload)`: unsafe JSON → typed JWT.
* `src/auth/OAuth2.res` — multiple, OAuth response + JSON parsing.
* `src/mcp/McpServer.res` — 10+, JSON param extraction + tool-result build; every
  `Obj.magic(args)["field"]` is an unchecked access.
* `src/compose/ComposeOrchestrator.res` — `Yaml.parse(content)->Obj.magic` (no
  validation); `Obj.magic(options)` for fetch.
* `src/policy/PolicyStore.res` — `Obj.magic` in policy code.

Plus: JWT/JTI revocation hardcoded `hasRevocationList = false // TODO`. (NOTE:
JWT *signature* verification was already wired on this branch — commit
`6863383`, closing issue #12 — so the residual gap is the revocation ledger and
the unchecked boundary casts, not signature checking.)
